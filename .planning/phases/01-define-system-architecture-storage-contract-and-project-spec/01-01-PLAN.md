<objective>
Define the architecture contract for the reusable Instagram content system.

Purpose: Establish the shared workflow boundaries, the project-spec contract, the deterministic storage layout, and the growth boundaries before implementation begins.
Output: Three durable design docs that later phases can implement against.
</objective>

<scope>
## In scope
- Shared-vs-project boundary for the reusable workflow.
- Markdown-based project-spec contract with fixed sections and order.
- Deterministic per-post storage contract.
- Growth policy for automatic follow/unfollow and likes, plus batch approval for replies.
- Syncing the planning state with the resolved decisions.

## Out of scope
- Implementing content generation.
- Implementing publishing or engagement automation.
- Building the reusable bootstrap package itself.
- Any project-specific brand content.
</scope>

<work_plan>
## Task 1 — Lock the shared architecture contract
- Finalize the system-wide description of the shared workflow.
- State clearly that branding lives outside the shared bootstrap.
- Document that the cron-fed idea source is upstream and not part of this phase’s implementation work.

## Task 2 — Finalize the project-spec contract
- Keep the template in Markdown with headings.
- Preserve the fixed order:
  1. Project name
  2. Target platform
  3. Brand voice
  4. Visual rules
  5. CTA conventions
  6. Growth strategy
  7. Approval behavior
  8. Prohibited angles
  9. Asset constraints
  10. Operational notes (optional)
- Make the approval section explicit about batch approval for replies.

## Task 3 — Finalize the storage contract
- Keep the default local root as `content/posts/`.
- Keep the folder naming convention deterministic: `YYYY.MM.DD - Short subject`.
- Require the artifact set: `manifest.json`, `brief.md`, `cards.json`, `caption.md`, `approval.md`, and `assets/`.
- Keep traceability fields in the manifest.

## Task 4 — Capture growth policy
- Mark follow/unfollow as automatic.
- Mark comment likes as automatic.
- Require batch approval for reply drafts before sending.
- Require rate limits or spread rules for automated activity.

## Task 5 — Synchronize planning state
- Update `.planning/STATE.md` so the latest decisions are explicit.
- Remove any stale open questions that were already answered.
- Keep `.planning/ROADMAP.md` aligned with the phase-1 decisions.
</work_plan>
