<objective>
Build the reusable bootstrap package for the shared Instagram content suite.

Purpose: provide a predictable onboarding path for new environments, install the shared scaffold, validate the setup, and keep all project-specific branding outside the bootstrap.
Output: a modular bootstrap flow with minimal core behavior, optional template/default support, and strong validation/dry-run behavior.
</objective>

<scope>
## In scope
- Define the bootstrap entrypoint and its lifecycle.
- Implement a minimal reusable scaffold for the shared skill suite.
- Add optional templates/defaults for new environments.
- Implement strong validation, including a dry-run mode.
- Ensure the bootstrap can consume an existing project spec rather than recreate brand decisions.
- Keep the bootstrap aligned with the shared-vs-project boundary from phase 1.
- Sync `.planning/STATE.md` and `.planning/ROADMAP.md` if any bootstrap decisions need to be recorded.

## Out of scope
- Content generation skills themselves.
- Project-specific branding content.
- Cron intake wiring.
- Growth automation beyond what the bootstrap needs to validate boundaries.
- Any phase-3 or phase-4 runtime behavior.
</scope>

<work_plan>
## Task 1 — Lock the bootstrap contract
- Define the bootstrap’s responsibility boundaries in the shared suite.
- Decide the smallest reusable command or entrypoint shape.
- Make sure the bootstrap consumes project-specific spec input rather than embedding brand decisions.
- Document the hybrid onboarding behavior: defaults first, prompts only when needed.

## Task 2 — Build the minimal scaffold
- Create the shared bootstrap core.
- Generate only the essential structural files and templates.
- Keep the scaffold modular so optional pieces can be added later without redesigning the core.
- Ensure the scaffold is reusable across different brands and projects.

## Task 3 — Add validation and dry-run support
- Validate the environment before making changes.
- Add a dry-run mode that shows what would happen without modifying anything.
- Verify the bootstrap is internally consistent and ready for downstream workflow skills.
- Make validation failures specific and actionable.

## Task 4 — Verify project-spec and contract compatibility
- Confirm the bootstrap can read an existing project spec.
- Confirm the bootstrap respects the storage and project-spec contracts from phase 1.
- Ensure the bootstrap does not hardcode brand content or project-specific behavior.
- Make sure later workflow skills can rely on the bootstrap output without extra modifications.

## Task 5 — Capture reusable operational details
- Record any stable bootstrap decisions in `.planning/STATE.md`.
- Keep `.planning/ROADMAP.md` aligned if the phase scope changes or clarifies expectations.
- Capture any bootstrap usage notes that will help later phases and future projects.
</work_plan>
