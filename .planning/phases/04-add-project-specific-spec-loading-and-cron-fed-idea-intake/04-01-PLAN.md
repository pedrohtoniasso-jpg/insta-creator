<objective>
Connect project-specific specs, cron-fed idea intake, and approval-gated growth workflows to the shared system.

Purpose: wire the reusable content workflow to per-project inputs and outbound engagement behavior without changing the shared core.
Output: project-spec loading, normalized cron intake, growth queueing with approval gates, and end-to-end traceability from idea to publication or engagement action.
</objective>

<scope>
## In scope
- Load a project-specific Markdown spec without changing the shared bootstrap.
- Validate and parse project-spec headings so brand rules remain human-readable and structured.
- Normalize cron-fed idea intake into a hybrid structured shape.
- Preserve traceability from noisy intake into the shared workflow.
- Add growth action queueing for automatic low-sensitivity actions and approval-gated reply drafts.
- Store traceability metadata from idea through the final engagement or publishable artifact.
- Add documentation and examples for project integration and growth ops.
- Update `.planning/STATE.md` and `.planning/ROADMAP.md` if new stable decisions are confirmed.

## Out of scope
- Rewriting the shared workflow core from phase 3.
- Changing the bootstrap package to own project branding.
- Direct publishing integrations that are not needed for the project-spec / intake / growth wiring.
- Pure content generation improvements unrelated to project-specific integration.
</scope>

<work_plan>
## Task 1 — Wire project-spec loading and validation
- Define how a project-specific Markdown spec is discovered and loaded.
- Keep the project spec external to the shared bootstrap and shared core.
- Validate the heading-based structure so project rules remain readable and dependable.
- Make sure project-specific rules are passed into the shared workflow as inputs rather than copied into the core.

## Task 2 — Normalize cron-fed idea intake
- Define the hybrid intake shape with free-form idea text plus structured fields.
- Normalize incoming cron ideas so the shared workflow can trace them reliably.
- Preserve source metadata, priority hints, and format hints without making cron the editorial authority.
- Keep the normalized intake compatible with later brief generation.

## Task 3 — Add growth queueing and approval gates
- Distinguish automatic actions from approval-gated actions.
- Keep follows, unfollows, and comment likes automatic where allowed by the project rules.
- Queue reply drafts for batch approval before sending.
- Preserve review state and decision history so the growth path remains auditable.
- Keep the growth flow aligned with the project-specific rules instead of hardcoding them in the shared workflow.

## Task 4 — Preserve traceability end to end
- Carry the same traceability chain from idea intake through brief, cards, caption, approval, and final action.
- Add lineage metadata to the integration points so the system can explain why an action exists.
- Ensure rejected or revised outputs can feed back into project documentation for later runs.
- Keep the artifact bundle and audit trail in sync.

## Task 5 — Capture reusable integration guidance
- Record confirmed integration decisions in `.planning/STATE.md`.
- Keep `.planning/ROADMAP.md` aligned with the phase scope and outputs.
- Add documentation or examples that show how a project-spec file and cron intake should be shaped.
- Note any adapter boundaries that should stay stable for future projects.
</work_plan>
